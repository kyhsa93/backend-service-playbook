# FastAPI Implementation

## Overview

[FastAPI](https://fastapi.tiangolo.com/) is a Python-based ASGI web framework with built-in Pydantic validation and `Depends()`-based DI.
The concrete guide and runnable examples implementing this playbook's principles with FastAPI live in `implementations/fastapi/` in this repo.

**→ [implementations/fastapi/CLAUDE.md](../../implementations/fastapi/CLAUDE.md)** — entry point for the detailed FastAPI implementation guide (keyword → `docs/architecture/*.md` index)
**→ [implementations/fastapi/docs/architecture/](../../implementations/fastapi/docs/architecture/)** — FastAPI-specific elaboration docs corresponding to the root's 21 topics (same structure as NestJS's `docs/architecture/`)
**→ [implementations/fastapi/examples/](../../implementations/fastapi/examples/)** — full implementation example of the Account/Card/Payment domains (account creation/deposit/withdrawal/suspend/reactivate/close + SES notifications, card issuance + Account↔Card integration, card payment/payment cancellation/refund + bidirectional Payment↔Account Integration Events + a Domain Service)
**→ [implementations/fastapi/harness/](../../implementations/fastapi/harness/)** — automated evaluator that checks guide compliance

This doc is a coverage mapping between the root principle docs and the FastAPI implementation docs. There's a `docs/architecture/*.md` corresponding to each of the root's 21 topics — except 2 topics (strategic-ddd, domain-service), which are language/structure-agnostic so no FastAPI-specific page was made for them (NestJS made the same call). cross-domain-communication is covered by [`cross-domain.md`](../../implementations/fastapi/docs/architecture/cross-domain.md).

**An important distinction**: the Coverage column in the table below judges "**does the doc explain the root principle accurately and thoroughly**." A doc being "Complete" doesn't mean the actual code in `examples/` follows that principle 100% — each doc explicitly states the code's current gaps in a "Known gaps" section, and the **Notes** column below also always notes it when there's a code gap.

- **Complete**: a new `docs/architecture/<topic>.md` was written that accurately explains the root principle in FastAPI/Python idiom (judged Complete even if the code doesn't yet fully follow the principle, as long as the doc states the gap and presents the correct pattern)
- **Partial**: a doc exists but is absorbed into another doc (no dedicated page) or the explanation is thin
- **Missing**: no FastAPI-specific doc exists, and the root doc is referenced as-is since the topic is language/structure-agnostic

## FastAPI-specific, no corresponding root doc

- Automatic request/response schema validation via Pydantic `BaseModel`/`EmailStr`, and automatic OpenAPI (Swagger) doc generation
- A `Depends()` functional DI graph — the dependency graph is composed from provider functions with no class decorators (`account_router.py`'s `_repo`, `_notification_service`)
- Swapping dependencies for the test environment via `app.dependency_overrides` (E2E tests replace `get_session` with a testcontainers session)
- `implementations/fastapi/docs/architecture/bootstrap.md` — `main.py`'s `FastAPI(...)` construction, `lifespan`, router/exception-handler registration, and how FastAPI's automatic OpenAPI (`/docs`) replaces NestJS's explicit `SwaggerModule.setup()`
- `implementations/fastapi/docs/architecture/cross-domain.md` — a FastAPI implementation example of the Adapter pattern (ACL): the two BCs Account/Card really exist, so it covers the ABC under `application/adapter/` + the implementation under `infrastructure/` + `Depends` binding as real code (see [cross-domain-communication.md](../architecture/cross-domain-communication.md) for the principle)
- `implementations/fastapi/docs/architecture/design-principles.md` — a summary of the 13 core design principles running through this repo's other 21 docs
- `implementations/fastapi/docs/architecture/module-pattern.md` — how, in FastAPI (which has no DI container or module decorators), the Python package tree and `Depends` factory functions replace NestJS's `@Module`/`providers`, plus resolving circular imports
- `implementations/fastapi/docs/architecture/rate-limiting.md` — a guide to rate limiting based on `slowapi` (actually implemented in `examples/`: global `default_limits` plus per-endpoint `@limiter.limit()` on write endpoints)
- `implementations/fastapi/docs/architecture/shared-modules.md` — placement rules for code that doesn't belong to any domain (`src/common/`, `src/config/`, `src/auth/`, `src/outbox/`, `src/database.py`) — really shared by all three Account/Card/Payment domains

## Why FastAPI

- Repository/Technical Service interfaces can be defined with nothing more than the Python standard library's `ABC`/`abstractmethod`, making it easy to keep Domain/Application purity without framework decorators (the harness's domain-purity check assumes this too).
- Pydantic unifies DTO validation and serialization into a single `BaseModel`, naturally supporting the principle of keeping the Interface layer thin (Interface DTO = thin wrapper).
- Native `async`/`await` support handles I/O-bound work — SQLAlchemy's `AsyncSession`, `aioboto3`, etc. — consistently asynchronously.
- `Depends()`-based functional DI makes Handler assembly explicit right in the router code, so you can trace the dependency flow just by reading the code, with no container magic.
