# Kotlin Spring Boot Implementation

## Overview

Kotlin Spring Boot is a JVM-based server that combines Spring's mature ecosystem (JPA, DI container, `@Transactional`) with Kotlin's null-safety, `data class`, and `sealed class`.
The concrete guide and runnable examples implementing this playbook's principles with Kotlin Spring Boot live in `implementations/kotlin-springboot/` in this repo.

**→ [implementations/kotlin-springboot/CLAUDE.md](../../implementations/kotlin-springboot/CLAUDE.md)** — entry point for the detailed Kotlin Spring Boot implementation guide (keyword → doc index)
**→ [implementations/kotlin-springboot/docs/architecture/](../../implementations/kotlin-springboot/docs/architecture/)** — a Kotlin-specific detailed doc for each of the root's 21 architecture topics, plus 6 Kotlin-specific bonus docs (27 files total)
**→ [implementations/kotlin-springboot/examples/](../../implementations/kotlin-springboot/examples/)** — full implementation example of the Account domain (open/deposit/withdraw/suspend/reactivate/close + SES notifications)
**→ [implementations/kotlin-springboot/harness/harness.sh](../../implementations/kotlin-springboot/harness/harness.sh)** — automated evaluator that checks guide compliance (9 checks, focused on structure, naming, and annotation placement)

This doc is a coverage mapping between the root principle docs and the Kotlin Spring Boot implementation docs. Each of the 21 files under the root's `docs/architecture/` has a corresponding Kotlin-specific doc. **The level of documentation and whether the actual code in `examples/` follows that principle are two separate things** — each doc explicitly calls out any remaining code gap as a "known gap," and the Notes column in the table below repeats that emphasis.

## Why Kotlin Spring Boot

- **Null-safety built into the type system**: nullability like `Account?` is enforced at compile time, so "not found" can be expressed without wrapping in Java's `Optional<T>` or writing runtime `NullPointerException` guards.
- **`data class` expresses immutable DTOs/VOs at zero cost**: Commands, Results, and Value Objects are all declared in one line with no Lombok or hand-written `equals()`/`hashCode()`/`toString()`, and immutable updates via `copy()` come naturally.
- **`sealed class`/`sealed interface` let the compiler verify type hierarchies**: sealing domain exceptions/events lets the compiler check exhaustiveness in `when` branches, catching missed handling of a newly added type at compile time.
- **100% interoperable with the Java Spring ecosystem**: you can use Java-ecosystem libraries like JPA, Spring MVC, and Testcontainers as-is while still gaining language-level safety from Kotlin — though points where Kotlin's defaults (classes are `final`) conflict with Spring AOP proxy requirements, like the need for `open class`/the `kotlin-spring` plugin, need to be managed separately.
