# Java Spring Boot Implementation

## Overview

[Spring Boot](https://spring.io/projects/spring-boot) is the de facto standard backend framework in the Java world, with Spring Data JPA, declarative transactions (`@Transactional`), and a DI container built in.
The guide and runnable examples implementing this playbook's principles with Spring Boot live in `implementations/java-springboot/` in this repo.

## harness.sh verification scope

`implementations/java-springboot/harness/harness.sh` consists of 8 sections (file-naming, repository-annotation, service-annotation, domain-purity, controller-placement, package-structure, shared-infra, event-placement) and only checks file placement, annotation position, and directory existence. Here are its limitations â€” things the harness does NOT catch:

- `package-structure` only checks whether directories exist â€” a violation of `cqrs-pattern.md`'s stated rule that "a Query Service must not use a Repository directly" isn't caught by this rule alone (the harness's separate `cqrs-query-purity` rule checks for this).
- `domain-purity` only forbids Spring stereotype annotations â€” **JPA annotations are not checked**. Even though `Account`/`Transaction`/`Money` are separated out as pure domain objects, if someone regresses by putting `@Entity` back on a domain class, this harness won't catch it.
- There's no check at all in the harness for things like Repository method naming, actual soft-delete behavior, or test-layer composition.

Even for an item the doc marks "Covered," the harness often doesn't substantially enforce it â€” verification mostly stays at the level of structural placement rules.

## Why Java Spring Boot

- Spring Data JPA (`JpaRepository`) and declarative `@Transactional`/`@Transactional(readOnly = true)` provide the Repository pattern and transaction propagation out of the box at the framework level.
- Transaction-propagation attributes can be controlled declaratively, e.g. `@Transactional(propagation = Propagation.REQUIRES_NEW)`, so a pattern isolating notification-send failures from propagating into the original command's transaction (`NotificationServiceImpl`) can be implemented with just an annotation.
- Java records (JDK 16+) let you express Value Objects, Commands, Results, and Query objects as concise immutable objects (`Money`, `AccountFindQuery`, `CreateAccountCommand`).
- Lombok (`@RequiredArgsConstructor`) cuts down constructor-injection boilerplate, keeping Application/Infrastructure-layer code that uses DI relatively concise.
- Spring Boot Actuator + Micrometer provide health checks/metrics out of the box at the framework level, so adopting graceful shutdown and observability needs almost no extra library combination.
