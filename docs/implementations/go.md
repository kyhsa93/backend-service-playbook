# Go Implementation

## Overview

Go is a per-language implementation of this playbook's principles using just the standard library (`net/http`, `database/sql`) plus a minimal set of third-party dependencies.
The guide and runnable examples implementing this playbook's principles with Go live in `implementations/go/` in this repo.

**→ [implementations/go/docs/architecture/](../../implementations/go/docs/architecture/)** — of the root's 24 `docs/architecture/*` docs, 21 have a corresponding Go implementation detail doc (Strategic Design is language-agnostic so the root doc is referenced as-is; for domain-service, the root doc already cites nestjs's code as its real example, so Go doesn't have a separate doc either, but the table below cites Go's own code; cross-domain-communication is covered by [`cross-domain.md`](../../implementations/go/docs/architecture/cross-domain.md))
**→ [implementations/go/examples/](../../implementations/go/examples/)** — full implementation example of the three Account/Card/Payment Bounded Contexts (account open/deposit/withdraw/suspend/reactivate/close + email notifications, card issue/suspend/close, card payment/payment cancellation/refund)
**→ [implementations/go/harness/](../../implementations/go/harness/)** — automated evaluator that checks guide compliance

This doc maps how much, and how, the Go implementation (21 `docs/architecture/*.md` docs + the `examples/` code + `harness/`) covers each of the root's 24 `docs/architecture/*` docs. **The level of documentation and whether the `examples/` code actually follows the principle are two separate things.** Where a doc explains the correct pattern while still noting a gap in the current code (e.g. UUID hyphens), that's left as-is in the Notes column below.

## Summary — remaining code gaps

Each of the 21 `implementations/go/docs/architecture/*.md` docs explains the root principle, cites the real `examples/` code wherever possible, and where the code doesn't yet fully follow the principle, states it as a "known gap" and presents the target implementation in code. Below are the gaps **still remaining in the current code**:

1. **persistence.md**: context-based transaction propagation across multiple Repositories — not built yet since there's no Command Handler that actually writes to two Aggregates' Repositories at once (YAGNI, the same reasoning as `shared-modules.md`'s explanation for the absence of `tx.go`). Migration rollback (`NNNN_*.down.sql`) is now resolved, with a down file added for every up file.
2. **file-storage.md**, **scheduling.md**: the pattern itself isn't implemented in `examples/` at all — the docs are entirely target designs.
3. **repository-pattern.md**: resolved — `account.Repository` has been unified into the single `FindAccounts(query)` method (a single-record lookup is `Take: 1` + the `FindOne` helper), and the absence of `Delete<Noun>` is still an intentional design (Account only transitions state via `Close()`).
4. **cross-cutting-concerns.md**: resolved — `middleware/logging_middleware.go` (`RequestLogging`) logs method/path/status code/elapsed time and is included in `router.go`'s middleware chain.
5. **observability.md**: resolved — `internal/infrastructure/logging.CorrelationHandler` wraps `slog.Handler` to auto-inject the ctx's correlation ID into every log record.
6. **error-handling.md** (shared with rate-limiting.md): resolved — the 429 response in `rate_limit_middleware.go` is now unified into the `{statusCode, code, message, error}` JSON schema (`RATE_LIMIT_EXCEEDED`).

## What the harness actually verifies / doesn't verify

`implementations/go/harness/main.go` (307 lines) only checks the following 7 structural/placement rules:

1. Whether file names are `snake_case.go`
2. Whether the `internal/{domain,application,infrastructure,interface}` + `application/{command,query}` directories exist
3. Whether the Repository interface is under `domain/` and its implementation under `infrastructure/`
4. Whether a Handler file is under `application/command|query/` or `interface/http/`
5. Placement of `*_task_controller.go` → `interface/`, `*_scheduler.go` → `infrastructure/`
6. If any outbox/task-queue-related file exists, whether a dedicated `outbox/`, `task-queue/` directory (e.g. `internal/infrastructure/outbox/`) exists somewhere under `internal/`
7. Placement of `*_event_handler.go` → `application/event/`, `*_integration_event.go` → `application/integration-event/`

**The harness purely checks file location and naming, and doesn't semantically verify any of the topics the 21 `docs/architecture/*.md` docs cover.** Here's what the harness does NOT catch:

- Whether Repository method naming follows the root rule (`find<Noun>s`/`save<Noun>`/`delete<Noun>`)
- Whether a CQRS Handler actually has the `Handle(ctx, cmd/query) (result, error)` signature
- Whether an error is a sentinel error, or whether the HTTP status code mapping follows the standard JSON schema
- Whether a soft-delete column actually exists and is filtered on lookup
- Whether an Aggregate ID is in the hyphen-free 32-char hex format (currently `examples/` follows the rule via `common.NewID()`, but the harness doesn't enforce this, so it would pass even on a regression)
- Whether Domain/Application unit test files exist

Expanding the harness rules to cover the items above is worth considering as a future improvement.

## Go-specific, no corresponding root doc

- **Compile-time interface-satisfaction verification** — the `var _ order.Repository = (*OrderRepository)(nil)` idiom. Unlike TypeScript's structural typing or NestJS's DI-token binding, Go gets static safety from this pattern — "if the implementation doesn't satisfy the interface, compilation itself fails." None of the root docs mention this kind of compile-time verification mechanism.
- **Package-level encapsulation** — Go has no instance-level `private` (see `tactical-ddd.md`). The root docs don't assume this constraint.
- [`bootstrap.md`](../../implementations/go/docs/architecture/bootstrap.md) — the dependency-assembly order against the real code in `cmd/server/main.go` (constructor chaining, no DI container), and a target `main()` shape integrating config/middleware/graceful-shutdown improvements.
- [`cross-domain.md`](../../implementations/go/docs/architecture/cross-domain.md) — Adapter pattern implementation details (see the root's [cross-domain-communication.md](../architecture/cross-domain-communication.md) for the principle). The two BCs Account/Card really exist, so it covers both the synchronous Adapter and the asynchronous Integration Event as real code.
- [`design-principles.md`](../../implementations/go/docs/architecture/design-principles.md) — a 14-item TL;DR checklist condensing the other 20 docs.
- [`module-pattern.md`](../../implementations/go/docs/architecture/module-pattern.md) — given that Go has no DI container, explains how manual constructor assembly and the package tree stand in for NestJS's `@Module`, and the advantage that the Go compiler blocks circular dependencies outright with no workaround like `forwardRef()`.
- [`rate-limiting.md`](../../implementations/go/docs/architecture/rate-limiting.md) — middleware based on `golang.org/x/time/rate`'s token bucket. A global limiter is really implemented in `examples/`, injected into `NewRouter`, and applied across all write routes — a per-client (IP) limiter map, differentiated per-route application, and exposing `X-RateLimit-*` headers remain unadopted extension options (forward-looking) from the doc. An over-limit 429 response follows the same `{statusCode, code, message, error}` JSON schema (`RATE_LIMIT_EXCEEDED`) as other error paths.
- [`shared-modules.md`](../../implementations/go/docs/architecture/shared-modules.md) — the placement convention for shared code like `internal/common/`, `internal/interface/http/middleware/`, etc. Explains that, unlike NestJS's `@Global` module declaration, "sharing" here just means `main.go` passes the same instance to multiple constructors.

## Why Go

- **Standard library first** — `go.mod`'s direct dependencies are just the AWS SDK, `google/uuid`, `lib/pq`, and `testify`/`testcontainers` for testing. No web framework, no ORM, no DI container.
- **The structural typing of `interface` fits naturally with Repository/Technical Service abstraction.**
- **CQRS's Handler pattern is expressed with just a struct + methods, no framework.**
- **Static compilation and explicit error handling (returning `error`) pair well with the Domain/Application layers' plain-error principle.**
